## 决策记录

### ADR-001: 将产品定位为能力校准与恢复工具

**日期**: 2026-06-11  
**状态**: 已采纳

**背景**

项目最初接近“个人外脑 + 技能树 + 轻量知识库”的组合。进一步澄清后，核心问题不是资料存储，而是跨领域学习者如何确认能力边界、提升抽象理解，并在一段时间后快速恢复实操能力。

**决策**

Sycamore 的核心定位定义为“能力校准与恢复工具”。节点以能力断言为中心，而不是以主题词条为中心。

**备选方案**

- 通用知识库：实现简单，但容易退化为轻量 Obsidian。
- 纯技能树：视觉直观，但难以表达跨领域关系和低频细节。
- LLM 学习助手：交互强，但容易让用户被动接受答案。

**后果**

产品边界更清晰，MVP 应围绕能力节点、心智模型、Cheatsheet、实践记录和 LLM 评审展开。代价是录入摩擦高于普通笔记，但这正是反囤积机制的一部分。

### ADR-002: MVP 采用 CLI-first 和 local-first 架构

**日期**: 2026-06-11  
**状态**: 已采纳

**背景**

项目未来可能需要 GUI 或 Web，但当前最需要验证的是学习闭环，而不是界面复杂度。

**决策**

MVP 采用 CLI 作为主入口，核心数据保存在本地 SQLite 和 Markdown 文件中。核心业务逻辑先沉淀为应用库，不先实现常驻 API 服务。

**备选方案**

- 直接做 Web：体验更直观，但会显著增加前后端复杂度。
- 直接做桌面 GUI：适合能力地图，但会推迟核心学习机制验证。
- C/S 架构起步：便于未来扩展，但对单人本地工具来说过早。

**后果**

实现路径更短，更符合单人项目和本地隐私需求。未来 GUI 可以作为 core 层的适配器加入。

### ADR-003: LLM 只作为评审者，不作为最终作者

**日期**: 2026-06-11  
**状态**: 已采纳

**背景**

项目目标是提升用户本身的能力，而不是构建一个更方便的答案查询器。如果 LLM 自动生成完整笔记，用户可能跳过主动编码，削弱学习效果。

**决策**

LLM 在 MVP 中只用于评审用户写下的 Mental Model。它可以指出错误、边界、类比风险和追问，但不能覆盖用户原文，不能标记知识为事实已验证。

**备选方案**

- 允许 LLM 生成完整节点：效率高，但削弱主动学习。
- 完全不接入 LLM：更纯粹，但缺少外部质疑机制。

**后果**

Prompt 和数据模型必须围绕 critique 设计。评审状态使用 `challenged`、`needs_revision`、`accepted_by_user`，禁止使用 `verified`。

### ADR-004: Markdown 正文优先，SQLite 作为索引

**日期**: 2026-06-11  
**状态**: 已采纳

**背景**

用户希望知识库轻量、非侵入、可迁移。若正文只存在数据库中，工具损坏后用户难以继续访问自己的知识。

**决策**

Markdown 文件是节点正文的主要载体，SQLite 保存元数据、路径、状态、关系和搜索索引。

**备选方案**

- 全部存 SQLite：查询方便，但降低可迁移性。
- 全部存 Markdown：简单透明，但关系和状态查询困难。

**后果**

必须处理双存储一致性问题，并提供 `syca doctor` 和索引重建能力。

### ADR-005: P0 采用 Capture-first，而不是 AbilityNode-first

**日期**: 2026-06-11  
**状态**: 已采纳

**背景**

前一版设计要求用户在录入时直接创建高质量 AbilityNode，填写能力断言、Mental Model、Cheatsheet 和 Practice Log。批判性审查后发现，这会在真实编码和排障场景中造成录入瘫痪。用户最需要系统时，往往只想快速保存一条命令或一个坑。

**决策**

P0 从 `capture` 开始，而不是从 `add AbilityNode` 开始。Sycamore 引入 `CaptureItem` 和 Inbox，允许低质量输入先进入缓冲区，再通过 `promote` 渐进式升格为正式 AbilityNode。

**备选方案**

- 继续强制 AbilityNode：能保证节点质量，但会导致使用阻力过高。
- 做普通剪藏 Inbox：摩擦低，但容易退化为资料垃圾箱。

**后果**

P0 命令调整为 `init/capture/inbox/promote/query/sync/doctor`。LLM Review、能力图、恢复演练后移到 P1/P2。反囤积原则通过“CaptureItem 不能自动冒充 AbilityNode”实现。

### ADR-006: P0 技术栈收敛为轻量本地 CLI

**日期**: 2026-06-11  
**状态**: 已采纳

**背景**

Sycamore 的首要风险不是扩展性不足，而是早期复杂度过高导致 P0 无法稳定使用。Web、ORM、RAG、常驻服务都会增加实现、测试和维护成本。

**决策**

P0 技术栈采用 Python 3.12+、uv、Typer、Rich、SQLite、sqlite3 repository 封装、python-frontmatter、TOML 和 pytest。不引入 FastAPI、SQLAlchemy/Alembic、向量数据库、Web 前端、后台 daemon 或插件系统。

**备选方案**

- FastAPI + Web 起步：扩展性强，但过早引入服务和前端复杂度。
- ORM 起步：抽象更完整，但 schema 未稳定时迁移成本偏高。
- 直接做语义检索：看起来智能，但会把产品导向查询数据库。

**后果**

P0 更容易实现、测试、备份和回滚。代价是早期需要手写少量 SQL 和 repository 封装，但这比引入 ORM 更符合当前阶段。

### ADR-007: promote 保留 UUID 并补充短升格入口

**日期**: 2026-06-11  
**状态**: 已采纳（P0.7 待实现）

**背景**

P0 端到端验收完成，但 `syca promote <capture-id>` 要求完整 UUID。用户在 `syca inbox` 与 promote 之间复制长 ID 的摩擦明显高于 capture / query 步骤，削弱 Capture-first 闭环的流畅度。

**决策**

保留完整 UUID 作为精确匹配方式，在 P0.7 增加更短升格入口：至少支持 `--latest`、`--index <n>`（对应 inbox 序号）、UUID 前缀唯一匹配；可选支持无参交互式选择。

**备选方案**

- 仅支持 UUID：实现最简单，但日常升格体验差。
- 完全改为序号主键：短，但 inbox 排序变化时易混淆。
- 仅交互式 TUI：体验好，但不利于脚本化和测试。

**后果**

CLI 契约需更新 `data-contracts.md`；inbox 列表应稳定排序（当前按 `created_at DESC`）；前缀匹配必须检测歧义并拒绝。
